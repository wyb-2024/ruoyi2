## 为什么这些改动能解决乱码
### 1. **使用 OkHttp 直接调用**
+ 避免了 Spring Boot 框架层面的编码转换
+ 直接控制请求和响应的编码方式

### 2. **明确指定 UTF-8 编码**
+ 在 MediaType 中指定 `charset=utf-8`
+ 在请求头中指定 `Content-Type: application/json; charset=utf-8`
+ 确保整个请求链路使用 UTF-8

### 3. **使用 charStream() 而非 byteStream()**
+ `byteStream()` 返回字节流，需要手动解码
+ `charStream()` 返回字符流，自动按正确编码解码
+ 避免了手动解码可能导致的编码错误

### 4. **注释调试代码**
+ 避免在控制台输出可能包含中文的调试信息
+ 防止控制台编码问题影响实际业务

## 乱码问题改动总结
### 一、后端配置文件改动
**文件路径：** `d:\ruoyi-ai\ruoyi-admin\src\main\resources\application.yml`

**改动位置：** 第 83-85 行（server.servlet.encoding 配置部分）

**改动内容：**

```yaml
server:
  servlet:
    encoding:
      charset: UTF-8
      enabled: true
      force: true
      force-response: true  # ✅ 新增：强制响应头使用 UTF-8 编码
```

**修改原因：**

1. **Spring Boot 版本升级导致的配置变更**
    - Spring Boot 3.4 版本中，`spring.http.encoding` 配置已被废弃
    - 新的配置路径为 `server.servlet.encoding`
    - 系统启动时会警告：

```plain
Key: spring.http.encoding.charset
Replacement: server.servlet.encoding.charset
```

2. **解决大模型回复乱码**
    - `charset: UTF-8`：设置字符编码为 UTF-8
    - `enabled: true`：启用编码配置
    - `force: true`：强制所有请求和响应使用 UTF-8 编码
    - `force-response: true`：**关键配置**，强制响应头使用 UTF-8 编码，确保返回给前端的数据正确编码

---

### 二、前端配置文件改动
**文件路径：** `d:\ruoyi-ai\ruoyi-web\vite.config.ts`

**改动位置：** server 配置部分（约第 30-40 行）

**改动内容：**

```typescript
server: {
  host: '0.0.0.0',
  port: 1002,
  open: false,
  headers: {  // ✅ 新增：设置响应头编码
    'Content-Type': 'text/html; charset=utf-8'
  },
  proxy: {
    // ... 其他配置保持不变
  },
}
```

**修改原因：**

1. **确保前端页面正确显示中文**
    - Vite 开发服务器默认可能不指定字符编码
    - 通过设置 `Content-Type: text/html; charset=utf-8` 响应头
    - 确保浏览器以 UTF-8 编码解析 HTML 内容
2. **与后端编码保持一致**
    - 后端强制使用 UTF-8 编码返回数据
    - 前端也明确指定 UTF-8 编码接收数据
    - 双重保障避免编码不一致导致的乱码
